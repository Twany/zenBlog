如果本地代理里有两个订阅，最难的通常不是“让某几个域名走另一条线路”。真正难的是：这套分流能不能在订阅刷新后继续成立，复杂服务的依赖域名会不会漏到错误出口，旧连接会不会残留，以及过几天以后你还能不能说清楚每条链路到底被谁用了多少流量。

所以这件事最后不应该写成一份配置记录。更自然的理解方式是：一个小小的分流需求，逐层逼出了一套本地网络治理流程。先让出口角色稳定，再让规则可恢复，再用观测数据证明它长期有效。

## 小需求背后的真实问题

表面需求很简单：主订阅 A 负责大多数流量，辅助订阅 B 只负责少数指定域名。比如 ChatGPT/OpenAI/Codex、部分 Apple 接口、Figma、Atlassian 这类对出口稳定性更敏感的服务。

但这个需求里藏着三个更实际的问题。

第一，出口要一致。某些服务并不只关心能不能连上，还会受到登录状态、风控、WebSocket、静态资源、接口请求等因素影响。如果同一个产品的不同请求从不同出口出去，就可能出现页面能打开、接口失败、流式响应中断、登录验证异常这类难排查的问题。

第二，订阅会变化。节点不是静态资产，订阅刷新后节点可能改名、增删、替换。手动复制节点或手动点选线路，短期能工作，长期一定会变成维护负担。

第三，实时状态不等于历史证据。Clash Verge 的连接页能告诉你“现在这条连接走哪里”，但它不能自然回答“过去一天有没有漏网”“哪几个进程最耗 B 链路”“30 天下来日志会多大”。

这三个问题合在一起，才是这套方案真正要解决的东西。

## 先把线路变成角色

双订阅分流要先定义角色，而不是先写规则。A 和 B 不应该只是两个节点池，而应该是两个职责不同的出口。

| 角色 | 负责什么 | 设计目的 |
| --- | --- | --- |
| 主订阅 A | 日常网页、下载、普通 API、原订阅自带规则 | 保持原来的默认体验，不扩大影响面 |
| 辅助订阅 B | 指定的敏感服务和依赖域名 | 保持这些服务的出口一致性 |
| 未归类流量 | 交回 A 的原规则 | 避免把未知流量误导到 B |

这个角色划分会影响后面所有实现。如果没有这层边界，后续很容易滑向两个极端：要么全局切到 B，影响太大；要么在 UI 里到处手动点节点，刷新一次就要重新排查。

一个可维护的分流结构应该长这样：

```text
主订阅 A
  ├─ 保留原本的 proxy-groups 和 rules
  ├─ 引入辅助订阅 B 的本地 profile/provider
  ├─ 建立一个只使用 B 节点的线路组
  └─ 把目标域名规则放在 A 原规则之前
```

这里最重要的一点是“引用”，不是“复制”。主订阅不应该保存一份 B 节点的静态副本，而应该引用 B 已经下载到本地的订阅 profile 或 provider。这样 B 刷新后，节点仍然由 B 自己更新；A 只维护“哪些域名应该导向 B”这层路由逻辑。

## 规则要经得起刷新，而不是只在今天生效

很多代理配置之所以越用越乱，是因为第一次解决问题时只追求“现在能用”。但订阅是动态的，浏览器连接是长连接，规则还有先后顺序。只要这些现实条件被忽略，配置就会在某个刷新或重连之后变回不可解释的状态。

这里有一个很实用的 before/after：

| 临时做法 | 可维护做法 |
| --- | --- |
| 把 B 的节点复制到 A 里 | A 引用 B 的本地 profile/provider |
| 在规则末尾补几条域名 | 把目标规则前置到 A 的原规则之前 |
| 手动刷新后凭记忆重配 | 用脚本或 Skill 重新生成结构 |
| 看到旧连接还在就继续怀疑规则 | reload 后关闭旧连接，让新连接重新命中规则 |
| 只适配当前 A/B | 把“主订阅/辅助订阅”做成可替换角色，未来 C/D 也能复用 |

这也是为什么这套分流最后适合做成 Codex Skill。用户不需要每次记住改了哪个文件、插了哪段规则，只需要确认两个角色：哪个订阅是主 A，哪个订阅是辅助 B。剩下的事情交给脚本：查找 profile、注入 provider、生成线路组、前置规则、重载配置、关闭旧连接、验证目标域名。

当一件事第二次被问到“下次怎么快速恢复”，它就已经不是配置问题，而是自动化问题。

## 漏网域名会把你带到“服务依赖组”

分流规则最容易低估的地方，是复杂服务并不等于一个主域名。

`chatgpt.com` 走 B，不代表 `chat.openai.com` 一定走 B。页面域名、WebSocket、静态资源、登录验证、错误上报、遥测、支付、挑战验证，都可能落在不同域名上。对于 OpenAI/Codex 这类服务，把其中一部分请求留在 A，另一部分导到 B，反而会制造更隐蔽的问题。

OpenAI 的 [网络建议](https://help.openai.com/en/articles/9247338-network-recommendations-for-chatgpt-errors-on-web-and-apps) 也能看出这一点：它列出的不是一个域名，而是一组 OpenAI/ChatGPT 相关域名，并且单独提到了 ChatGPT 和 Codex 的 WebSocket 需求。这说明分流不能只按“首页域名”理解，而要按“服务依赖组”管理。

更稳的处理方式是：

1. 先把主域名和官方建议里的基础域名放进 B 规则组。
2. 用当前连接和历史采样观察还有哪些相关域名仍然走 A。
3. 把漏网域名补进同一个依赖组。
4. reload 后关闭旧连接，观察新连接是否重新命中 B。
5. 把这套补漏过程沉淀进脚本或 Skill，而不是只补一次。

这个方法也适用于 Figma、Atlassian、Apple 等服务。你不必一开始就猜中所有域名，但必须有一套能持续发现漏网域名的机制。

## 实时连接页不够时，看板才有意义

做到这里，分流已经能工作。但只靠 Clash Verge 的连接页，仍然只能回答“此刻发生了什么”。一旦连接刷新、浏览器重连、页面关闭，历史就散了。

真正需要看板的时刻，是当问题从“现在走哪条线路”变成这些：

- 某个域名过去几个小时有没有偶尔走错？
- B 链路今天到底用了多少流量？
- 哪些进程在使用 B，是浏览器、编辑器，还是后台服务？
- 某条规则是不是经常误命中？
- 新补的 OpenAI/Codex 规则，长期看是否稳定？
- 采样文件保留 30 天大概会占多少空间？

这时候看板不是装饰，而是证据系统。它把瞬时连接变成可回看的历史，把“我感觉走对了”变成“我能看到过去一段时间都走对了”。

看板的交互也应该顺着排查路径设计，而不是把原始连接一股脑倒出来。最有用的路径是：先看链路，再点开某条链路看进程汇总，再看域名和规则。

| 看板视角 | 它回答的问题 |
| --- | --- |
| 链路汇总 | A/B 各用了多少上传和下载 |
| 链路下钻进程 | 某条链路主要被哪些进程消耗 |
| 域名汇总 | 哪些目标站点消耗最多 |
| 规则汇总 | 哪些规则命中最多 |
| 时间段筛选 | 问题发生在哪个时间窗口 |

这种设计比“实时连接列表加强版”更有用，因为它直接服务于排查：先缩小到链路，再缩小到进程，再缩小到域名。

## 一个完整走通的例子

把前面的逻辑连起来，一个自然的处理流程大概是这样。

原始需求是：OpenAI/Codex 相关请求需要稳定走 B，其他流量继续走 A。第一步不是去找某个节点，而是确认 B 是这个服务组的稳定出口。第二步是在 A 的规则前面插入 OpenAI/Codex 依赖组，让这些域名在 A 的原规则之前命中 B。第三步是 reload 配置，并关闭旧连接，避免浏览器长连接继续显示旧链路。

接着进入验证阶段。先看当前连接：目标域名是否新建到了 B。再看历史采样：过去一段时间有没有相关域名仍然留在 A。如果发现 `chat.openai.com`、某个静态资源域名或 WebSocket 目标仍然在 A，就不是“配置失败”，而是依赖组还没补全。把它补进 B 规则组，再 reload、关闭旧连接、继续观察。

最后才进入长期维护。把这套动作写进 Skill，未来换成新的 C/D 订阅时，只需要重新确认主订阅和辅助订阅；把采样器做成后台进程，网页关掉也继续记录；给日志加保留周期，避免工具越跑越占空间。

这就是从一次分流配置到长期流量审计的完整闭环。

## 采样器必须独立于网页

如果看板要承担历史审计，它的采样不能依赖网页是否打开。网页只是展示窗口，collector 才是事实来源。

更合理的结构是拆成两部分：

```text
Mihomo / Clash Core
        |
        | /connections API
        v
collector  --->  data/traffic-YYYY-MM-DD.jsonl
        |
        v
.run/collector-status.json

web dashboard  --->  读取 JSONL + collector status
```

collector 定时读取 Mihomo `/connections`，计算上传和下载增量，按天写入 JSONL。web dashboard 只读取这些历史文件，按链路、进程、域名、规则和时间段聚合展示。

这个拆分带来一个重要好处：网页服务停了，采样不应该停。你可以关掉浏览器，甚至暂时不启动看板，collector 仍然在后台积累数据。等需要排查时，再打开网页查看过去的历史。

在 macOS 上，长期后台进程更适合交给 LaunchAgent。Apple 的 [Launch Daemons and Agents 文档](https://developer.apple.com/library/archive/documentation/MacOSX/Conceptual/BPSystemStartup/Chapters/CreatingLaunchdJobs.html) 说明了 `launchd` 如何管理后台任务、用户级 agent、`ProgramArguments` 和 `KeepAlive` 等配置。实际使用时，再用一层 shell 脚本提供 `start`、`stop`、`status`、`logs` 这种简单入口，日常操作就不必记 Node 命令和进程号。

这里还有一个细节值得记住：纯 collector 模式下，Node 的采样定时器不能 `unref()`。如果定时器被 unref，而进程里没有 HTTP server 之类的其他保活任务，Node 会认为没有必须等待的工作，然后正常退出。后台采样器需要一个明确的生命周期锚点。

## 验证链路，也要知道看板的边界

当看板显示 B 链路有流量时，还需要能回答一个更底层的问题：它是否真的到达了预期的服务器？

这个问题最好从两端验证。本机侧，确认辅助节点的 server 和端口，再看本机 Clash/Mihomo 进程是否建立了到这个地址的 TCP 连接。服务器侧，用 `ss -tnp` 查看是否有来自本机侧的已建立连接，服务端进程是否是代理服务。

两侧都能看到连接，才能比较确定链路是通的。至于云厂商控制台流量为空，不一定说明没有走到服务器，也可能是统计延迟、流量太小、按 GB 四舍五入、控制台统计口径不同，或者账单口径和 Mihomo 应用层统计口径不同。

这也提醒我们：本地看板适合回答“哪些进程、域名、规则走了哪条链路，以及 Mihomo 观测到多少应用层流量”。它不是云厂商账单系统，不能直接拿来对账。

## 日志保留是长期工具的一部分

一旦 collector 常驻，日志保留就不是附加功能，而是生命周期设计的一部分。

比较稳的方式是按天写文件，例如 `traffic-YYYY-MM-DD.jsonl`，默认保留 30 天。collector 启动时清理一次，之后每小时清理一次。需要更短周期时，可以把保留天数改成 7 或 14；需要临时排查时，也可以单独调整。

空间估算不要靠猜。直接看当前采样文件在一两个小时里增长多少，折算成每天大小，再乘以保留天数。这样得到的不是绝对准确值，但足够决定 7 天、14 天还是 30 天。

这一步看起来不像分流配置，却决定了工具能不能长期存在。没有保留策略，任何采样器最后都会变成磁盘问题。

## 可以直接复用的检查表

如果你也要做类似的双订阅分流和流量看板，可以按这个顺序检查：

1. 先定义出口角色：默认出口是谁，稳定出口是谁。
2. 不复制节点，用 profile/provider 引用动态订阅。
3. 把目标域名规则放到主订阅原规则之前。
4. 对复杂服务按依赖组管理，而不是只写首页域名。
5. reload 后关闭旧连接，让新连接重新命中规则。
6. 把重复恢复动作做成脚本或 Skill，只让用户确认 A/B 角色。
7. 用实时连接页解决当前问题，用历史采样验证长期稳定性。
8. 看板按“链路 -> 进程 -> 域名 -> 规则”设计下钻路径。
9. collector 和 web 分离，采样进程必须独立常驻。
10. 用 LaunchAgent 或 systemd 托管后台进程。
11. 从本机和服务器两侧验证链路，不只相信一个界面。
12. 给日志设置保留周期，并用真实采样量估算磁盘占用。

## 结语

这件事的自然流程，其实不是“先做分流，再做看板”。更准确地说，是一个问题不断暴露下一个问题。

你先想让少数域名走 B，于是需要定义出口角色。你发现订阅会刷新，于是需要把配置变成可恢复结构。你发现主域名不够，于是需要按服务依赖组补规则。你发现实时连接页只能看现在，于是需要历史采样。你发现网页关了不能影响采样，于是需要后台 collector。你发现日志会增长，于是需要保留策略。

所以最终沉淀下来的不是某几条规则，而是一套可迁移的方法：先让结构稳定，再让过程自动化，再用观测证明结果，最后补齐长期运行的生命周期。
