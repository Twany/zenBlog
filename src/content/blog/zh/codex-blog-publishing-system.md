真正值得沉淀的不是 Codex 最后改出来的那几行代码，而是这中间的判断：为什么这样做、哪些路没有走、哪些约束需要保留、最后怎么验证。我的做法是把“发布到博客”做成一个明确的工作流：触发后直接生成中英双语长文、跑校验、部署上线；只有我明确说“先写草稿”时，系统才停在草稿状态。

## 为什么我需要这个系统

我用 Codex 做了很多实际工作：改 UI、查可访问性问题、修构建工具链、把 Header 样式推到 Cloudflare 上看效果，又根据真实观感回退。每一次看起来都是“完成一个任务”，但里面其实藏着很多可写的内容。

比如这次博客 UI 调整，真正值得写的不是“把 Header 改轻了又恢复了”，而是几个更有复用价值的问题：

- 让 AI 改界面时，应该先给审美要求，还是先给设计约束？
- 一个博客的可读性问题，为什么不能只看颜色和间距？
- AI 生成内容时，怎样避免一键上线带来的失控？
- 如果我希望每次 Codex 工作都能变成文章，项目本身需要提供什么基础设施？

以前这些想法会散落在对话里。任务做完后，如果不马上整理，过几天就只剩一个模糊印象。于是我决定把内容沉淀接到工作流末端，而不是等以后有空再回忆。

## 核心判断：触发语本身就是发布确认

我一开始把这个系统设计成“先草稿、再确认发布”。这个边界很安全，但实际用起来会多一个中断点。后来我意识到，如果触发语本身已经足够明确，比如我说的是“发布到博客”，那它就不是普通沉淀，而是一次发布指令。

所以最终规则改成：默认直接发布，但发布前仍然要经过内容安全检查和工程校验。草稿能力保留，只在我明确说“先生成草稿”“先别发布”时使用。

所以这个系统的边界很明确：

| 环节 | 可以自动化什么 | 保留什么边界 |
| --- | --- | --- |
| 提炼主题 | 从对话中找问题、决策和方法 | 只提炼可公开的经验 |
| 写作 | 生成中文长文和英文版本 | 不能编造事实，不能泄露私密信息 |
| 写入博客 | 生成 Markdown、frontmatter、slug、阅读时间 | 默认 `draft: false`，除非明确要求草稿 |
| 校验 | 跑内容 schema、lint、Astro check、build | 任一步失败就停止发布 |
| 部署 | 执行 Cloudflare deploy 并检查线上页面 | 部署后必须验证 URL |

这张表就是整套系统的原则：触发语给出发布意图，脚本负责把发布路径变成可靠流程。

## 这次我是怎么把它做进博客的

这不是一个独立的写作工具，而是嵌进当前博客项目的一套能力。博客本身是 Astro 项目，文章存在 `src/content/blog/zh` 和 `src/content/blog/en` 两个目录里，前者是中文，后者是英文。Astro 的 Content Collections 本来就适合这类内容，因为它可以用 schema 约束 frontmatter，并在构建时提供类型检查和内容查询能力。这个判断也和 Astro 官方对 [Content Collections](https://docs.astro.build/en/guides/content-collections/) 的定位一致。

我做的第一件事，是给文章 schema 增加 `draft` 字段。它最初是默认草稿，后来改成“草稿例外”：正常发布时写入的是：

```yaml
draft: false
```

同时保留查询逻辑：本地开发环境可以看到 `draft: true` 的草稿，生产构建会过滤草稿。这样我仍然有一个安全阀，但默认的“发布到博客”路径不会停在草稿审阅。

第二件事，是把写文件这一步做成脚本。以前如果让 AI 直接生成 Markdown 文件，它每次都要重新处理 slug、translationKey、readTime、frontmatter 格式，错误空间很大。现在脚本只接收一个 JSON：

```json
{
  "slug": "codex-blog-publishing-system",
  "category": "AI Workflow",
  "tags": ["Codex", "AI Writing", "Automation"],
  "zh": {
    "title": "中文标题",
    "description": "中文描述",
    "body": "中文正文"
  },
  "en": {
    "title": "English title",
    "description": "English description",
    "body": "English body"
  }
}
```

脚本负责生成两篇 Markdown，自动计算阅读时间，保证中英文共用同一个 slug 和 translationKey。这一步把“写作质量”和“文件格式”拆开了：Codex 主要负责内容，脚本负责重复、容易出错的结构化工作。

第三件事，是把发布做成另一个脚本。它会确保中英文两篇文章都是 `draft: false`，然后依次跑 lint、Astro check、build，最后部署到 Cloudflare。以后触发“发布到博客”时，这一步会直接执行；只有草稿例外流程才会停下来等待审阅。

这里也参考了 Cloudflare Workers Static Assets 的工作方式：Wrangler 会把配置中指定目录的静态资源上传到 Cloudflare；Cloudflare 文档里也说明了这种 [Static Assets](https://developers.cloudflare.com/workers/static-assets/) 部署模型。我的项目已经有 Wrangler 配置，所以发布脚本只需要把构建和 deploy 串起来。

## Before / After：这套系统改变了什么

| 以前 | 现在 |
| --- | --- |
| 工作完成后，靠记忆判断有没有内容价值 | 触发“发布到博客”后，先从对话里提炼可公开的方法 |
| AI 可能直接写一篇文章，但格式和路径要重新解释 | Skill 固定项目路径、schema、分类、标签和脚本入口 |
| 中文和英文容易变成两套不一致的内容 | 中文先写，英文做地道重写，共用 translationKey |
| 发布和写作混在一起，容易误上线 | 触发语明确时直接发布；草稿只作为显式例外 |
| 验证靠临时想起要跑什么命令 | 脚本固定执行 lint、check、build，再部署 |
| 内容里可能混入本机路径或账号细节 | Skill 明确要求抽象敏感信息，只保留公开经验 |

这张对比表里最重要的变化是“意图明确”。当我说“发布到博客”时，系统不再假装这是普通笔记整理，而是按发布流程完整走完；当我说“先写草稿”时，它才退回审阅模式。

## 为什么必须是长文，而不是随笔

我的博客里本来有长文和随笔，但这个 Skill 只负责长文。原因很简单：随笔更像个人感受，应该由我自己写；Codex 更适合沉淀流程、案例和方法。

一次 Codex 工作如果值得发布，通常应该能回答几个问题：

- 原始问题是什么？
- 我为什么这样指挥 AI？
- AI 具体做了什么？
- 中间遇到了什么限制？
- 哪个判断可以复用到下一次？
- 如果别人照做，最容易踩的坑在哪里？

这些问题天然适合写成长文。长文不只是篇幅更长，而是有完整的推理链：从场景到决策，从决策到实现，从实现到验证。

## 中文先写，英文不是逐句翻译

我把默认语言定成中文，因为原始思考通常发生在中文里。中文稿先确定观点、结构和案例，英文稿再重新组织成地道表达。

这一步很关键。如果英文只是逐句翻译，很多中文里的表达会变得生硬。更好的方式是保持同一个主张，但允许英文版本调整句式、段落和标题。

比如中文标题可以是：

> 把 Codex 的工作过程变成博客：我给自己做了一个“发布到博客”系统

英文标题不必直译成 “Turn Codex's working process into a blog”。更自然的写法是：

> Turn Codex Work into Blog Posts: The Publishing System I Built for My Own Blog

这不是语言能力的炫技，而是面向不同读者的重新表达。

## 可复用的检查清单

如果我要把这套方法迁移到另一个内容站，我会按这个顺序做：

1. 先确认内容模型：文章放在哪里，frontmatter 有哪些必填字段，路由怎么生成。
2. 增加草稿字段：保留 `draft: true` 作为例外安全阀，生产构建必须过滤草稿。
3. 固定写入脚本：不要让 AI 每次手写重复结构，让脚本处理 slug、readTime、translationKey。
4. 明确触发语：比如“发布到博客”，不要让普通对话误触发发布流程。
5. 明确默认行为：触发“发布到博客”就直接发布；触发“先写草稿”才停在草稿。
6. 在 Skill 里写质量门槛：必须有具体案例、对比表、检查清单，不能有敏感信息。
7. 发布前跑校验：内容 schema、lint、check、build 至少要过。
8. 部署后验证线上地址：不要只相信 deploy 成功日志。

如果少了第 2 步，系统很容易误发；如果少了第 3 步，每次都会重新犯格式错误；如果少了第 6 步，文章很容易变成空泛的 AI 总结。

## 常见问题

### 这种系统会不会让博客变得很“AI 味”？

会，前提是你只让 AI 总结结果，不要求它复盘判断。我的解决方式是强制文章包含 raw input、决策过程、实现取舍和验证结果。只写“做了什么”很像流水账，写清“为什么这样做”才有人的判断。

### 直接发布会不会太冒险？

会，如果触发语含糊，或者没有校验和安全检查。但我的触发语本身就是发布指令，系统还会在发布前检查敏感信息、内容结构、Astro schema、lint 和 build。风险控制不一定意味着多一次人工确认，也可以意味着让发布路径足够明确、失败时自动停止。

### 为什么要做英文版？

英文版不是为了显得国际化，而是为了逼自己把方法讲得更通用。很多工作流如果只能在中文语境里成立，说明它还没有被抽象干净。英文重写会暴露结构上的含糊之处。

### 这套方法只适合技术博客吗？

不只适合技术博客。只要内容来自可复用的工作过程，比如产品决策、增长实验、设计复盘、自动化搭建，都可以用类似方式沉淀。区别只在于内容 schema 和发布脚本。

## 结语

这次我真正想解决的不是“让 AI 帮我写博客”，而是让日常工作不要白白流失。Codex 每次帮我完成任务时，都会留下判断、取舍和方法。把这些东西变成文章，需要的不是随手一键生成，而是一条可验证、可失败中止、可直接上线的发布路径。

我现在有了一个明确入口：说“发布到博客”，系统就会生成中英双语长文、通过校验、部署上线。只有我明确说“先写草稿”，它才停下来给我审。

这才是我希望的 AI 内容生产方式：AI 负责整理过程和执行发布链路，人负责定义触发边界和内容方向。
